## PHASE 1: KILL THE FAKES (Week 1-2)

> Goal: Remove everything that lies. No more pretend AI.

### 1.1 -- Destroy the Fake Model Router

**File:** `lib/langraph/model-router.ts` (203 lines)

**Problem:** 7 categories, every one routes to `openrouter/auto`. 177 lines of copy-paste.

**Fix:**
- Replace with a real router: simple queries -> Gemini Flash, complex reasoning -> GPT-4o, code -> Claude, creative -> GPT-4o
- Delete the entire switch statement. Replace with a ~40-line function
- Use `classifyPrompt()` output to actually select different models
- Map: `chat`/`creative` -> Gemini Flash (fast, cheap), `reasoning`/`analysis` -> GPT-4o, `code` -> Claude Sonnet, `action`/`retrieval` -> Gemini Flash
- Remove `openrouter/auto` as primary. Use specific models

### 1.2 -- Kill the Dead Reasoning Engine

**File:** `lib/nova/reasoning-engine.ts`

**Problem:** Regex pipeline with empty `MentalModel`. Zero reasoning.

**Fix:**
- Either: (a) Delete entirely and route directly to LangGraph agent, OR (b) Rewrite to actually populate `MentalModel` from DB and use it downstream
- **Recommendation:** Delete. The reasoning should happen inside the LLM via the LangGraph agent, not in a pre-processing regex step. Route intent classification into the agent's system prompt instead
- Update `app/api/ai/route.ts` to remove the `ReasoningEngine.reason()` call. Use `DecisionFramework.evaluate()` directly for risk gating, then pass intent context into the agent

### 1.3 -- Fix or Delete the Multi-Agent System

**File:** `lib/nova/agent-framework.ts`

**Problem:** Keyword matching with empty params. 2 of 8 agents non-functional.

**Fix:**
- Option A (recommended): Delete `agent-framework.ts` entirely. The LangGraph agent already handles tool calling with context. The "agent" system adds nothing
- Option B: If you want specialized agents, wire them into LangGraph as conditional node paths. But this is over-engineering for now
- Remove `orchestrate_agentic_workflow` inline tool from `lib/ai-tools/index.ts`
- Remove dead agent routes from `app/api/ai/agents/route.ts`

### 1.4 -- Fix the Hardcoded Tools

**Files:** `lib/ai-tools/index.ts`

**Problem:** `evaluate_risks` returns same 3 strings. `generate_standup` has hardcoded blockers. `generate_dashboard_config` returns static JSON.

**Fix:**
- `evaluate_risks`: **Delete entirely.** Risk analysis is the LLM's job via the agent. The tool should not return hardcoded data
- `generate_standup`: Remove hardcoded `["None reported by system"]`. Query blocked tasks from DB
- `generate_dashboard_config`: **Delete.** It returns static JSON. The dashboard should be rendered client-side, not generated by AI
- Remove any other tools that return hardcoded data

### 1.5 -- Fix Fake Tools in Task Tools

**File:** `lib/ai-tools/task-tools.ts`

**Problem:** `set_recurring` prepends text to description. `create_approval_request` creates notification, not workflow.

**Fix:**
- `set_recurring`: Either implement actual recurring task logic (scheduled job that creates duplicate tasks on interval) OR remove the tool entirely and tell the user "recurring tasks are not yet supported"
- `create_approval_request`: Either implement an approval state machine (status transitions, approver assignment, deadline, escalation) OR remove the tool

### 1.6 -- Wire the Constitution or Delete It

**Files:** `lib/nova/constitution/` (21 files), `lib/nova/config.ts`

**Problem:** `buildSystemPrompt()` exists but `app/api/ai/route.ts` uses `NOVA_SYSTEM_PROMPT` from config.ts instead.

**Fix:**
- Option A (recommended): Wire `buildSystemPrompt()` into the agent. Use it in `nova-agent.ts` when constructing the system prompt. This gives you intent-aware prompts (lighter for chat, heavier for actions)
- Option B: Delete the constitution system entirely. Just use the simpler prompt in config.ts
- **Recommendation:** Wire it. The constitution has good behavioral rules. Just don't send ALL sections on every request -- use `buildSystemPromptForIntent()` to send only what's needed

### 1.7 -- Delete Dead Mem0 Read Path

**File:** `lib/nova/memory-system.ts`

**Problem:** Writes to Mem0, never reads from it.

**Fix:**
- Either implement `getLongTerm()` to read from Mem0 with semantic search (embed the query, search Mem0's vector store)
- OR remove Mem0 entirely and just use Redis + Prisma
- **Recommendation:** Remove Mem0 for now. Add it back when you have vector embeddings set up properly

## PHASE 2: MAKE THE AI ACTUALLY INTELLIGENT (Week 2-4)

> Goal: Real intelligence, not regex pretending to be smart.

### 2.1 -- Real Proactive Intelligence

**File:** `lib/nova/proactive-intelligence.ts`

**Problem:** Hardcoded thresholds. "Launch" dependency detection.

**Fix:**
- Remove all magic numbers. Make thresholds configurable per workspace
- Replace "launch" dependency detection with: tasks with 0 subtasks AND estimated effort > some threshold AND status != "done"
- Add real stall detection: tasks where `updatedAt` is > 3 days ago AND `status == "in_progress"` AND have assignees
- Add duplicate detection: tasks with similar titles (Levenshtein distance or token similarity) in the same project
- Add capacity imbalance: if one member has > 3x the task count of the average
- Run this analysis via the LLM, not regex. Feed the data into the agent and let it reason about what's actually concerning

### 2.2 -- Real Memory with Semantic Retrieval

**File:** `lib/nova/memory-system.ts`

**Problem:** No semantic search. Hardcoded confidence 0.8.

**Fix:**
- Add embeddings: when saving long-term memory, generate an embedding via OpenAI `text-embedding-3-small` and store it
- When retrieving, embed the query and do cosine similarity search against stored embeddings
- Use Upstash Redis Vector or Prisma with a vector extension for storage
- Remove hardcoded `confidence: 0.8`. Calculate based on memory age, frequency of access, and recency
- Add memory consolidation: merge similar memories, prune stale ones

### 2.3 -- Real Multi-Step Planning

**File:** `lib/langraph/nodes/agent-planner.ts`

**Problem:** Detection is regex on "then". Plans are fragile.

**Fix:**
- Use the LLM itself to detect multi-step requests. Add a classification step: "Is this request complex enough to need planning?"
- If yes, have the LLM generate a numbered plan, then execute each step sequentially via the tool executor
- Remove the regex `/then|and then|steps/i` detection

### 2.4 -- Improve the LangGraph Graph

**Files:** `lib/langraph/agent-graph.ts`, `lib/langraph/nova-agent.ts`

**Problem:** Not using StateGraph properly. 500-line monolithic `execute()`.

**Fix:**
- Decompose `execute()` into actual LangGraph nodes:
  - `classifyIntent` node
  - `loadContext` node
  - `loadMemory` node
  - `evaluateRisk` node
  - `routeToAgent` node (conditional edge: simple -> direct LLM, complex -> tool-calling loop)
  - `toolExecutor` node (LangChain agent loop)
  - `qualityGate` node
  - `saveMemory` node
- Use LangGraph's `StateGraph` with `Annotation.Root` for state management
- Add conditional edges based on intent and risk level
- This makes the system debuggable, testable, and extensible

### 2.5 -- Reduce System Prompt Bloat

**Files:** `app/api/ai/route.ts`, `lib/nova/config.ts`

**Problem:** 3,000-5,000+ tokens of system prompt per request.

**Fix:**
- Use `buildSystemPromptForIntent()` from the constitution to send only relevant sections
  - CHAT intent: identity + philosophy (500 tokens max)
  - ACTION intent: identity + execution principles + security (1,000 tokens max)
  - ANALYSIS intent: identity + reporting standards + context (1,000 tokens max)
- Remove "future evolution" sections from prompts (aspirational docs mislead the LLM)
- Add workspace context dynamically, not statically. Only include context relevant to the current request
- Target: < 2,000 tokens of system prompt for most requests

## PHASE 3: COMPETE ON REAL FEATURES (Week 4-6)

> Goal: Features that competitors have that Nova doesn't.

### 3.1 -- Real Proactive Monitoring (Background Worker)

**Problem:** Nova only analyzes when explicitly called. Competitors run continuously.

**Fix:**
- Use Inngest (already installed) to create a scheduled function that runs every 15-30 minutes
- The function loads workspace data, runs proactive intelligence analysis, and creates notifications for:
  - Overdue tasks approaching deadlines
  - Blocked tasks with no recent updates
  - Capacity imbalances across team members
  - Sprint scope creep
- Store insights in a new `ProactiveInsight` model in Prisma
- Surface these in the Nova UI proactively (not just when asked)

### 3.2 -- Real Risk Prediction

**Problem:** Nova has no predictive capabilities. Wrike has ML-based delay forecasting.

**Fix:**
- Build a simple velocity tracker: calculate average task completion rate per team member over last 30 days
- Compare current sprint scope against historical velocity
- Use the LLM to generate risk assessments based on: velocity trends, dependency chains, team capacity, historical delays
- Store risk scores per project and surface them in analytics
- This doesn't need ML models -- just historical data + LLM reasoning

### 3.3 -- Smart Automation Rules

**Problem:** `automation-intelligence.ts` translates NL to automation but it's basic regex parsing.

**Fix:**
- Use the LLM to parse natural language into structured automation rules
- Define a clean schema: `{ trigger: { type, conditions }, actions: [{ type, params }] }`
- Let the LLM generate this schema from user input, then validate with Zod
- Store and execute via Inngest
- This is what Monday.com does with their AI Blocks

### 3.4 -- Real Meeting Intelligence

**File:** `lib/nova/meeting-intelligence.ts`

**Problem:** Template-based, not real transcription.

**Fix:**
- Integrate with a transcription API (AssemblyAI, Deepgram, or Whisper)
- Accept audio files or meeting URLs
- Generate summaries, extract action items, identify decisions
- Auto-create tasks from action items
- Store meeting notes linked to projects/tasks

### 3.5 -- Real Document Intelligence

**File:** `lib/nova/document-intelligence.ts`

**Problem:** Basic document operations, no understanding.

**Fix:**
- When a document is uploaded/created, generate embeddings for semantic search
- When the LLM needs to reference documents, do vector search to find relevant ones
- Add document summarization, key point extraction, action item detection
- Link document insights to tasks/projects

## PHASE 5: COMPETITIVE POSITIONING (Week 8-10)

> Goal: What Nova can do that differentiates it.

### 5.1 -- MCP Server (Model Context Protocol)

**Why:** Every competitor (ClickUp, Monday, Wrike, Notion, Smartsheet) now has an MCP server. This lets external AI tools (Claude, ChatGPT, custom agents) read/write workspace data.

**Fix:**
- Implement an MCP server that exposes Nova's workspace data
- Allow external tools to query tasks, projects, documents, and team data
- This is a major differentiator -- it makes Nova an "AI platform" not just an "AI tool"

### 5.2 -- AI-Powered Sprint Planning

**Why:** Linear and ClickUp have this. Nova has a Sprint Agent defined in constitution but never implemented.

**Fix:**
- Use the LLM to analyze backlog, team velocity, and dependencies
- Generate sprint plans with capacity-aware assignment
- Predict completion dates based on historical data
- Auto-detect scope creep and suggest adjustments

### 5.3 -- Smart Notifications

**Why:** Competitors have proactive alerts. Nova has passive notification storage.

**Fix:**
- Nova should generate contextual notifications: "Task X is at risk because it depends on Task Y which is overdue"
- Notifications should include AI-generated recommendations, not just "you were assigned"
- Use the proactive intelligence engine to drive this

### 5.4 -- AI Standup Reports

**Why:** ClickUp Brain does this well. Nova has `generate_standup` but it returns hardcoded data.

**Fix:**
- Actually generate standup reports from real data: what was completed, what's in progress, what's blocked
- Use LLM to summarize activity logs and task status changes
- Format as bullet points per team member
- Auto-generate daily and send via Slack/email integration
